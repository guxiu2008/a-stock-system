需求文档：A 股行情自动化同步工具 (market_quota_syncer.py)1. 任务目标构建一个稳健的 Python 脚本，利用 Tushare Pro API 抓取全量 A 股及主流指数的历史至今行情数据。核心要求是计算复权价格，并支持增量同步，为后续 Agent 的波动率分析和长线决策提供准确的数据底座。2. 核心逻辑：为什么要“复权”？在长线投资中，高送转和分红会导致股价“断崖式”下跌（除权）。如果不计算复权因子，Agent 会错误地认为股票发生了暴跌。计算公式： $复权价格 = 原始收盘价 \times 复权因子$建议方案： 采用前复权 (Forward Adjustment)，即以当前价格为基准，回溯历史价格。这样 Agent 看到的最新价格与交易软件一致。3. 详细功能需求A. 数据抓取范围个股日线 (daily)： 全量 A 股（参考 asset_registry 中的代码列表）。复权因子 (adj_factor)： 必须抓取每一只股票每天的复权因子。指数日线 (index_daily)： 包含上证指数 (000001.SH)、深证成指 (399001.SZ)、沪深 300 (000300.SH)、创业板指 (399006.SZ) 以及中证 1000。B. 同步模式设计全量模式 (Full Sync)： 首次运行时，下载过去 10-20 年（根据 Tushare 积分权限）的所有历史数据。增量模式 (Incremental Sync)： 每日 18:00 后运行，检查本地数据库中每只股票的“最后更新日期”，仅下载缺失日期的数据。由于 A 股有 5000+ 股票，必须严格遵守 Tushare 的每分钟访问频率限制（限流控制）。字段对齐： 确保计算后的 adj_close (前复权收盘价)、adj_open、adj_high、adj_low 准确存入数据库。量能处理： 保留 vol (成交量) 和 amount (成交额)，用于 Agent 识别“底部放量”信号。4. 存储架构建议 (针对你的 MongoDB/PostgreSQL 环境)推荐表/集合结构： fact_daily_quotes字段名类型说明索引建议ts_codeString股票代码 (例: 000001.SZ)复合索引 (1)trade_dateDate/String交易日期 (例: 20260327)复合索引 (2)closeFloat原始收盘价-adj_factorFloat当日复权因子-adj_closeFloat前复权收盘价 (计算得出)必填volFloat成交量 (手)-pct_chgFloat当日涨跌幅-5. 开发者注意事项（避坑指南）限流保护 (Rate Limiting)： Tushare 针对不同积分有每分钟调用次数限制。脚本必须包含 time.sleep() 或动态限流逻辑，防止 IP 被封。停牌处理： 如果某天股票停牌，Tushare 的 daily 接口可能没有数据。脚本需要能识别跳过的日期，不要在计算时报错。数据完整性检查： 每月建议进行一次“对账”，随机抽查几只股票的复权价与主流金融终端（如东方财富/同花顺）是否一致。指数不复权： 注意，指数（如沪深 300）通常不需要 adj_factor，直接存储 index_daily 即可。6. Agent 决策接口预留脚本完成后，需要为 OpenClaw 的 Agent 提供如下 Python 函数：get_history_prices(ts_code, start_date, end_date)：获取一段连续的复权行情。calculate_drawdown(ts_code, window)：计算过去 X 天的最大回撤。detect_volume_surge(ts_code)：识别最近 5 日成交量是否显著超过过去 20 日均值（底部放量逻辑）。下一步建议：这个脚本的数据量会很大（5000 股 x 240 个交易日/年），建议优先考虑使用 PostgreSQL 存储这部分行情数据（查询效率比 MongoDB 高），而 MongoDB 留给 asset_registry。你需要我提供一个基础的 Tushare 访问与复权计算的代码模板给你的开发人员参考吗？
